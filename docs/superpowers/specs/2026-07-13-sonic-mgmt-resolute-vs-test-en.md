## 1. Background & Goal

The SONiC resolute kernel/base-package migration shipped: `target/sonic-vs.bin` built on 2026-07-13, `os-release` confirms Ubuntu 26.04, kernel `7.0.0-1002-sonic` (Launchpad PPA), ~30 package migration fixes merged on the `202605_resolute` branch. The `docker-sonic-mgmt.gz` test-framework container (pytest 9.1.1) was built the same day and verified loadable/runnable.

Prior migration validation covered only a **KVM boot smoke test** (uname, 12 core containers Up). **No control-plane / config-pipeline regression check has been run against the sonic-mgmt test framework.** This design defines that first real check.

**Goal:** Drive the sonic-mgmt pytest suite from the `docker-sonic-mgmt` container against the resolute `sonic-vs` KVM DUT, running a **neighbor-independent** sanity subset to confirm the resolute migration (kernel + base packages + ~30 package fixes) did **not regress container health, the config pipeline, show commands, system services, or dataplane interfaces**.

**Non-goals:** Full T0 topology (requires Arista EOS neighbor VMs); BGP/routing tests; performance/scale; multi-ASIC.

```

   ‚îÇ  ansible / pytest                            SSH + config push
```

The executor reaches the DUT over the mgmt network via SSH/ansible to push config and collect show output. The PTF container (Phase 2) connects through an OVS bridge to the DUT's front-panel ports to send/receive test traffic.

## 4. Phased Execution

Design principle: **low-friction, high-value first; high-friction later.** Phase 1 covers the dimensions most likely affected by the migration and needs no traffic plane; Phase 2 runs only if Phase 1 is green and time permits.

### Phase 1 ‚Äî Core sanity (no PTF, 4 tests)

1. **Boot DUT:** `virsh create platform/vs/sonic.xml` (`sonic-vs.img` already in place); wait for 12 core containers Up + mgmt DHCP.
2. **Clone sonic-mgmt** @ `202605` to `~/sonic-mgmt` (with `https_proxy`).
3. **Configure executor:** start a container from the already-built `docker-sonic-mgmt` image (has sonic-mgmt deps + 3 dataplane/ptf patches): `docker run --network host -v ~/sonic-mgmt:/data docker-sonic-mgmt`. Two inventory paths, decided in the plan:
   - (a) directly hand-write a minimal inventory (pointing at the KVM DUT's IP/credentials); or
   - (b) run sonic-mgmt repo's `setup-container.sh` on the host and let it manage container/inventory setup (note: the standard `setup-container.sh` flow **creates** the sonic-mgmt container, whereas we already have a self-built image ‚Äî must make it reuse, not rebuild).
4. **Write minimal inventory** pointing at the KVM DUT (virsh default-net IP + admin/admin creds) ‚Äî if (b) above, generated by `setup-container.sh`.
5. **`testbed-cli.sh deploy-cfg`** to load the t0 minigraph (single-ASIC KVM hwsku `x86_64-kvm_x86_64-r0`).
6. **Run tests** (no PTF, `--neighbor-type none` semantics):
   - `test_features` ‚Äî container / feature state
   - `test_config_reload` ‚Äî config save / reload pipeline
   - `test_show_interface` ‚Äî interface show
   - `test_sonic` ‚Äî system-level

### Phase 2 ‚Äî PTF traffic (only if Phase 1 green + time permits)

1. Build `target/docker-ptf.gz` (`make target/docker-ptf.gz`, with proxy).
2. `testbed-cli.sh deploy-topo` to bring up the OVS bridge + PTF cabling.
3. Run `test_interfaces` (interface counters / flapping).

## 5. Pass Criteria

- **Phase 1:** all 4 tests PASS = no control-plane / config-pipeline regression in the resolute build.
- **Phase 2:** `test_interfaces` PASS = dataplane interface counters normal.
- On FAIL: apply `systematic-debugging` to distinguish a **resolute migration regression** from a **sonic-mgmt test itself / testbed-config issue** (see baseline below).

## 6. Baseline Comparison

When a test FAILs and the root cause is unclear, run the same test on the `~/sonic-buildimage-202605-clone` `sonic-vs.bin` (non-resolute, trixie baseline) as an A/B control to isolate the resolute factor. That clone has a sonic-vs.bin dated 2026-07-03 (bootability to be confirmed ‚Äî it exists but has not been booted in this session).

## 7. Risks & Mitigations

- **Testbed friction:** inventory credentials, unstable KVM mgmt IP (DHCP), minigraph port-count mismatch. Mitigation: simplify progressively ‚Äî first run only `test_features` (topology-independent) to confirm the link, then expand.
- **Baseline bootability:** whether `202605-clone`'s sonic-vs.bin boots cleanly needs verification (present but not yet booted here).
- **proxy dependency:** cloning sonic-mgmt and pip installs need `https_proxy=http://192.168.1.210:6152`.
- **Phase 2 cabling:** OVS bridge / port binding / minigraph port mapping are error-prone; Phase 2 failure does not invalidate the Phase 1 conclusion.

## 8. Local Environment Constraints (inherited from handoff)

- Working-tree local add-host / proxy edits in `Makefile.work` / `slave.mk` / `rules/config.user` are **not committed** (local acceleration only).
- Test-related code fixes (e.g. resolute-specific test adjustments) **may be committed but not pushed** (until green).
- Building PTF (Phase 2) reuses the cached slave image, with proxy.

- Phase 2 (if executed): `test_interfaces` report.
