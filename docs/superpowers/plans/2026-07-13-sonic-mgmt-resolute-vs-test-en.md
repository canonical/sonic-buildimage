# sonic-mgmt resolute VS Sanity Test — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Run a neighbor-independent sonic-mgmt sanity subset against the resolute `sonic-vs` KVM DUT to confirm the resolute migration did not regress container health, the config pipeline, or show commands.

**Architecture:** Boot the resolute `sonic-vs.img` as a KVM DUT (mgmt via QEMU user-mode networking, SSH on host port 3040). Clone `sonic-net/sonic-mgmt` @ `202605`. Build a minimal inventory + testbed definition pointing at `127.0.0.1:3040`. Run four neighbor-free pytest modules from inside the `docker-sonic-mgmt` container. Phase 2 (PTF) is out of scope for this plan.

**Tech Stack:** KVM/QEMU + libvirt (DUT), `docker-sonic-mgmt` (pytest 9.1.1 executor), sonic-mgmt ansible+pytest framework @ branch `202605`.

## Task 1: Boot the resolute sonic-vs DUT and verify SSH

**Files:**
- Create: `~/kvm-sonic-resolute/sonic.xml`
- (no test file — verification is the SSH login)

**Interfaces:**
- Consumes: `~/sonic-buildimage-resolute/sonic-vs.img` (already built), `~/sonic-buildimage-resolute/platform/vs/sonic.xml` (template)
- Produces: a running KVM domain `sonic` reachable at `ssh -p 3040 admin@127.0.0.1`, console at `telnet 127.0.0.1 7000`

- [ ] **Step 1: Prepare a working DUT boot XML**

```bash
mkdir -p ~/kvm-sonic-resolute
cp ~/sonic-buildimage-resolute/platform/vs/sonic.xml ~/kvm-sonic-resolute/sonic.xml
# Rewrite the hardcoded disk path to the resolute image
sed -i "s#/data/sonic/sonic-buildimage/target/sonic-vs.img#$HOME/sonic-buildimage-resolute/sonic-vs.img#" ~/kvm-sonic-resolute/sonic.xml
grep 'source file' ~/kvm-sonic-resolute/sonic.xml   # confirm path now points at /home/sheldon-qi/sonic-buildimage-resolute/sonic-vs.img
```

- [ ] **Step 2: Ensure no stale `sonic` domain, then create it**

```bash
sudo virsh destroy sonic 2>/dev/null; sudo virsh undefine sonic 2>/dev/null
sudo virsh create ~/kvm-sonic-resolute/sonic.xml
sudo virsh list --name | grep '^sonic$'   # expect: sonic
```
Expected: `sonic` listed as running.

- [ ] **Step 3: Wait for boot via console**

```bash
# Console is telnet 127.0.0.1 7000. Wait for login prompt.
( echo ""; sleep 90; ) | telnet 127.0.0.1 7000 2>/dev/null | tail -20
```
Expected: output contains `sonic login:`. If not, wait longer (SONiC first boot ~2-3 min to bring up containers).

- [ ] **Step 4: Verify SSH on host port 3040**

```bash
sshpass -p admin ssh -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null -p 3040 admin@127.0.0.1 'show version; show platform summary; docker ps --format "{{.Names}}" | wc -l'
```
Expected: SONiC version output + platform `x86_64-kvm_x86_64-r0` + container count `12` (the core containers). If SSH refused, the VM is still booting — re-run Step 3/4 after another 60s.

- [ ] **Step 5: Confirm resolute identity (this is the migration check point 0)**

```bash
sshpass -p admin ssh -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null -p 3040 admin@127.0.0.1 'cat /etc/os-release | grep -E "PRETTY|VERSION_CODENAME|ID="; uname -r'
```
Expected: `PRETTY_NAME="Ubuntu 26.04 LTS"`, `VERSION_CODENAME=resolute`, `ID=ubuntu`, kernel `7.0.0-1002-sonic`.

- [ ] **Step 6: No commit** (DUT XML is an artifact, lives outside the repo). Record the working SSH command in the test report note.
